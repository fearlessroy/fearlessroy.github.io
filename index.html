Mybatis一级缓存

发表于 2018-04-01 | 分类于 数据库

Mybatis一级缓存

Mybatis 提供了一级缓存方案来优化在数据库会话中重复查询的问题。实现方式是每个 SqlSession 中都持有自己的缓存。

一级缓存有两种级别：

  • SESSION 级别，在一个 Mybatis 会话中执行的所有语句，都会共享这个缓存。
  • STATEMENT 级别，缓存只对当前执行的 statement 有效。

每个 SqlSession 中拥有自己的 Executor，每个 Executor 维护一个 LocalCache， 发起查询时，Mybatis 会根据当前执行的 statement 生成一个 key，去 LocalCache 中查询，如果缓存命中，返回缓存中内容。如果缓存没有命中，查库，将结果写入 LocalCache，最后返回结果。

发表于 2018-03-31 | 分类于 数据库

最左前缀原理

全列匹配
按照索引中所有列进行精确匹配（这里精确匹配指“=”或“IN”匹配）时，索引可以被用到。这里有一点需要注意，理论上索引对顺序是敏感的，但是MySQL的查询优化器会自动调整where子句的条件顺序以使用适合的索引。
最左前缀匹配
当查询条件精确匹配索引的左边连续一个或几个列时，注意是连续，索引可以被用到。查询条件用到了索引中列的精确匹配，但是中间某个条件未提供，只会用到最左匹配的索引。

MySQL索引实现

发表于 2018-03-31 | 分类于 数据库

MySQL索引实现

在MySQL中，索引属于存储引擎级别概念。

MyISAM索引实现

MyISAM引擎使用B+Tree作为索引结构，叶节点的data域存放数据记录地址。
MyISAM索引文件仅仅保存数据记录地址
MyISAM中索引检索的算法为首先按照B+Tree搜索算法搜索索引，如果指定的Key存在，则取出其data域的值，然后以data域的值为地址，读取相应数据记录。
MyISAM辅助索引与主索引在结构上没有区别，只是主索引要求key唯一，辅助索引key可以重复。

数据库索引

发表于 2018-03-31 | 分类于 数据库

索引的本质

MySQL官方对索引的定义：索引(Index)是帮助MySQL高效获得数据的数据结构。
索引本质：一种数据结构。
查询数据是数据库的主要功能之一，最基本的查找算法是顺序查找，时间复杂度为O(N)，在数据量很大的情况下，这种算法的效率就会很低。那我们可以使用二分查找、二叉树查找等高效的算法，但是这些算法对检索的数据都有要求，如二分查找要求数据有序。但是数据本身不可能完全满足这些要求，所以，在数据之外，数据库系统维护着满足特定算法的数据结构，这些数据结构以某种方式引用数据，这样就可以在这些数据结构上实现高级查找算法，这种数据结构就是索引。

数据库软删除策略

发表于 2018-03-25 | 分类于 数据库

正常来说数据库记录不会直接硬删除(直接delete掉)，而是使用软删除，如在数据库表中增加一个字段，is_deleted，最简单的设置：0为未删除，1为已经被删除。

如果表中有UNIQUE字段col，使用软删除可能导致之后相同的字段值无法插入，因为这个字段在表中已经被使用且是唯一的，那这时候应该定义联合索引唯一约束即 UNIQUE KEY col_INDEX(col,is_deleted)。

不过这样会出现新的问题，col被第二次软删除，is_deleted为1的也被占用，进一步解决方法是把is_deleted定义为0未删除，非0表删除，非0可能是一个自增ID或者一个随机的值如时间戳、UUID等，那这样就能保证唯一性。
